# Atomic Knowledge Graph for Project Memories — Design Spec

## Goal

Replace single-blob project memory files with a folder-based atomic note structure that is
parseable by both Claude (at session start) and a human in Obsidian, follows the LYT MOC
pattern, and updates differentially — only writing atoms when the session contains material
new information.

---

## Vault Structure

```
second-brain/
  inbox/
    memories/
      index.md                        ← MOC of MOCs (auto-generated each run)
      ytk/
        index.md                      ← project hub (links only, auto-generated)
        purpose.md                    ← what it is and why — rarely changes
        tech.md                       ← tech stack with wikilinks — changes on new deps
        recent.md                     ← this session's work — always overwritten
        state.md                      ← what works / in-progress / blocked — changes on milestones
        questions.md                  ← open questions and blockers — changes when resolved
      epicmap/
        index.md
        purpose.md
        tech.md
        recent.md
        state.md
        questions.md
      ...
  projects/
    ytk/
      session-008-brief.md            ← episodic record, append-only, never modified
  sources/
    youtube/ · web/ · instagram/ · journal/
  wiki/
    hot.md · index.md
```

## Atom Definitions

Each atom is a standalone `.md` file with YAML frontmatter and a single focused section.

| Atom | Content | Update policy |
|---|---|---|
| `purpose.md` | One sentence: what the project is and why it exists | Only when purpose materially shifts |
| `tech.md` | Bulleted list of tools/libraries with wikilinks and per-project usage notes | Only when new tech introduced or removed |
| `state.md` | Working / In-progress / Blocked bullet lists | Only on significant milestone or regression |
| `questions.md` | Bullet list of unresolved decisions and blockers | Only when questions answered or new ones arise |
| `recent.md` | What happened this session — tools, commands, decisions, outcomes | **Always overwritten** |

### Atom YAML frontmatter

```yaml
---
type: atom
atom: purpose          # purpose | tech | state | questions | recent
project: ytk
updated: 2026-04-19
---
```

### Atom content templates

**purpose.md**
```
{project} is a [what] that [does what] for [whom/why].
```

**tech.md**
```
- [[tool-name]] — how used in this project specifically
- [[tool-name]] — ...
(max 8 items)
```

**state.md**
```
- Working: ...
- In progress: ...
- Blocked: ...   (omit if none)
```

**questions.md**
```
- ? [unresolved question or blocker]
(write `_no signal_` if none)
```

**recent.md**
```
- [what was done]
- [key decision or finding]
- [outcome or next step]
```

## Project Hub (index.md)

The hub contains only wikilinks — no prose. It is always regenerated by the script.

```markdown
---
type: project-hub
status: active
tech: [python, chromadb, anthropic-sdk]
last_active: 2026-04-19
---

## Current Understanding
[[purpose]] · [[tech]] · [[state]] · [[questions]]

## This Session
[[recent]]

## Session Log
- [[../../../projects/ytk/session-008-brief]] — 2026-04-19
- [[../../../projects/ytk/session-007-brief]] — 2026-04-18
```

`status` is one of: `active` | `paused` | `archived`. The script infers this from session
recency: active if last session < 14 days ago, paused if < 90 days, archived otherwise.

## MOC of MOCs (memories/index.md)

Auto-generated after every run. Contains only wikilinks grouped by status.

```markdown
# Projects

## Active
- [[ytk/index|ytk]] — personal knowledge system
- [[epicmap/index|epicmap]] — field mapping app

## Paused
- [[niloc/index|niloc]] — VIO research

## Archived
- [[hash/index|hash]] — concurrent Rust hash table
```

The one-liner after each link comes from the project's `purpose.md` (first sentence, truncated).

## Differential Update Algorithm

```
for each project:
  1. Read all existing atom files (may not exist on first run)
  2. Read session JSONL (--max-sessions N most recent)
  3. Call Haiku with: existing atoms + session excerpts → structured JSON
  4. JSON shape: { atom_name: { changed: bool, content: str | null } }
  5. `recent` is always { changed: true, content: ... }
  6. Write only atoms where changed == true
  7. Always regenerate project hub (updates last_active + session log links)

after all projects:
  8. Regenerate memories/index.md MOC from all project hubs
```

First-run behavior: all atoms are missing, so all are treated as `changed: true`.

## Haiku Prompt Design

Principles drawn from Claude Code prompt analysis:
- **Template over prose** — model fills named slots, not free-form text
- **Explicit sentinel** — `_no signal_` when content cannot be derived from session
- **Peer-accountability tone** — "Do not infer. Do not hallucinate."
- **Scoped authority** — each atom has its own strict template; the model cannot exceed it
- **Grounded in specifics** — "name tools, commands, and decisions concretely"

```
You are updating the knowledge base for the project '{project_display}'.

Below are the EXISTING atomic notes and excerpts from the most recent Claude Code session.

TASK: For each atom, decide whether the session introduces materially new information
that warrants updating it. If yes, return the complete updated content following the
template exactly. If no, mark it unchanged.

RULES:
- Follow each atom's template exactly. Do not add extra sections or prose.
- Be specific: tool names, CLI commands, decision rationale. Never vague summaries.
- Use [[wikilink]] syntax for tool names in tech.md.
- `recent` MUST always be updated — it reflects what happened in this session.
- For all other atoms: only update if the session introduces genuinely new information.
  A session about an unrelated topic does not update purpose, tech, state, or questions.
- If a section cannot be filled from actual session content, write `_no signal_`.
  Do not infer. Do not hallucinate.

ATOM TEMPLATES:

purpose — one sentence maximum:
  "{project} is a [what] that [does what] for [whom/why]."

tech — one bullet per tool, max 8:
  "- [[tool-name]] — [how used in this project specifically]"

state — working/in-progress/blocked:
  "- Working: [what functions correctly]"
  "- In progress: [what is partially built]"
  "- Blocked: [what is blocked and why]  (omit entirely if nothing blocked)"

questions — bullet list of open questions:
  "- ? [unresolved question or decision]"
  (write `_no signal_` if the session contains no open questions)

recent — what happened this session (ALWAYS required):
  "- [action taken]"
  "- [decision made or finding]"
  "- [outcome or immediate next step]"

---

EXISTING ATOM CONTENT:
{existing_atoms_block}

SESSION EXCERPTS:
{session_excerpt}

---

Respond with valid JSON only. No markdown wrapper. No explanation.
{
  "purpose":   { "changed": false } | { "changed": true, "content": "..." },
  "tech":      { "changed": false } | { "changed": true, "content": "..." },
  "state":     { "changed": false } | { "changed": true, "content": "..." },
  "questions": { "changed": false } | { "changed": true, "content": "..." },
  "recent":    { "changed": true,  "content": "..." }
}
```

## Scope

**Files modified:**
- `ytk/vault.py` — add `write_atom()`, `read_atom()`, `write_project_hub()`, `write_memories_moc()`
- `scripts/seed_memory.py` — new prompt, JSON output, differential update, folder writes

**Not in scope:**
- `vault_remember` MCP tool — unchanged (still writes flat ad-hoc notes to `inbox/memories/`)
- Session brief writing — unchanged
- All other vault write functions
- ChromaDB indexing schema — atoms are indexed by `reindex_vault` automatically via scan_dirs

**Migration:**
- Existing flat `project-*.md` files in `inbox/memories/` are deleted after the new structure
  is written on first run. The next seed run creates the folder structure from scratch.
